# User Story - US_013

## Story ID
* US_013

## Story Title
* Admin user account lifecycle management

## Description
* As an Admin, I want to create, edit, deactivate, and reactivate Staff and Admin user accounts with assigned roles, so that I can control who has operational access to the platform without requiring direct database intervention.

## Acceptance Criteria

### AC-001: Admin creates a new Staff account and credential setup email is sent
- **Given**: An Admin is authenticated and `POST /admin/users` is called with `{name, email, role:"Staff"}`
- **When**: The email does not already exist in UserAccounts
- **Then**: HTTP 201 is returned with `{"userId":"..."}` ; a UserAccount is created with `status=Inactive`, `setupPending=true`; a credential setup email containing a setup token is dispatched within 60 seconds

### AC-002: Admin updates a user's name, role, or status
- **Given**: An Admin is authenticated and a target UserAccount exists
- **When**: `PATCH /admin/users/{id}` is called with one or more of `{name, role, status}`
- **Then**: HTTP 200 is returned; only the specified fields are updated in UserAccounts; an AuditLog entry is written with before/after values

### AC-003: Admin cannot deactivate the last active Admin account
- **Given**: Only one UserAccount with `role=Admin` and `status=Active` exists
- **When**: `PATCH /admin/users/{id}` is called with `{status:"Inactive"}` for that account
- **Then**: HTTP 409 is returned with `{"error":"Cannot deactivate last active Admin account"}`; the UserAccount status is not changed

### AC-004: Duplicate email rejected on user creation
- **Given**: A UserAccount with `email='staff@clinic.com'` already exists
- **When**: `POST /admin/users` is called with the same email
- **Then**: HTTP 409 is returned with `{"error":"Email already registered"}`; no new row is inserted

### AC-005: Non-Admin cannot access admin user management endpoints
- **Given**: A Staff-role JWT is present
- **When**: `POST /admin/users` or `PATCH /admin/users/{id}` is called
- **Then**: HTTP 403 is returned; the request is logged in AuditLog with `actionType=READ` and a role-violation note

## Edge Cases
* Admin attempts to change their own role — the service prevents self-role-change with HTTP 409 `{"error":"Cannot modify your own role"}`
* Setup email delivery fails — the UserAccount is still created; a retry Hangfire job is enqueued; Admin sees a dashboard warning "Setup email pending retry"
* Concurrent Admin deactivation requests — the "last Admin" guard uses a SELECT COUNT with FOR UPDATE (or equivalent) to prevent a TOCTOU race
