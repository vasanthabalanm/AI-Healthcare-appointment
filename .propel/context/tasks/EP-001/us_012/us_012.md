# User Story - US_012

## Story ID
* US_012

## Story Title
* Patient self-registration with email verification

## Description
* As a patient, I want to register an account by providing my name, date of birth, contact details, and email address and receive a verification email, so that I can create a verified patient profile and access the appointment booking system.

## Acceptance Criteria

### AC-001: Successful registration sends verification email and returns 201
- **Given**: No UserAccount exists with the provided email address
- **When**: `POST /auth/register` is called with valid `{name, dob, phone, email, password}`
- **Then**: HTTP 201 is returned with `{"message":"Verification email sent"}`; a UserAccount row is inserted with `emailVerified=false`, `status=Inactive`, `role=Patient`; a verification email is dispatched within 60 seconds containing a single-use token

### AC-002: Email verification activates the account
- **Given**: A UserAccount exists with `emailVerified=false` and a valid unexpired verification token
- **When**: `GET /auth/verify-email?token={validToken}` is called
- **Then**: HTTP 200 is returned; `UserAccount.emailVerified` is set to `true` and `status` to `Active`; the token is invalidated; the response includes `{"redirect":"/login"}`

### AC-003: Duplicate email registration returns 409
- **Given**: A UserAccount already exists with `email = 'test@example.com'`
- **When**: `POST /auth/register` is called with the same email
- **Then**: HTTP 409 is returned with `{"error":"Email already registered","hint":"Login or reset password"}`; no new UserAccount row is inserted

### AC-004: Expired verification token returns 400
- **Given**: A verification token was issued more than 24 hours ago
- **When**: `GET /auth/verify-email?token={expiredToken}` is called
- **Then**: HTTP 400 is returned with `{"error":"Verification token expired"}`; the account remains in `status=Inactive`

### AC-005: Verification token resend issues a new token
- **Given**: A patient's verification token has expired
- **When**: `POST /auth/resend-verification {email}` is called with a valid registered email
- **Then**: HTTP 200 is returned; a new verification email is sent with a fresh 24-hour token; the old token is invalidated

## Edge Cases
* Registration with a password that does not meet complexity requirements — HTTP 422 returned with field-level validation errors; no UserAccount is created
* Email address with unicode characters — normalised to lowercase ASCII before uniqueness check and storage to prevent case-variant duplicates
* Verification link clicked twice — second click returns 400 `{"error":"Token already used"}`; account remains Active from first use
* Registration flood from same IP — rate limiting (10 registrations per IP per hour) returns HTTP 429 after threshold
