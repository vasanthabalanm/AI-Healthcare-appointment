# User Story - US_004

## Story ID
* US_004

## Story Title
* Upstash Redis and Hangfire background job infrastructure

## Description
* As a developer, I want Upstash Redis configured as the cache-aside layer and Hangfire 1.8.x registered with a SQL Server-backed job store, so that all session validation, slot caching, and background job processing have a ready-to-use, resilient infrastructure from day one.

## Acceptance Criteria

### AC-001: Redis connection established from environment variable
- **Given**: The `Redis:ConnectionString` environment variable is set to a valid Upstash Redis URL
- **When**: The application starts and a health check pings Redis
- **Then**: The health check returns `{"redis":"Healthy"}` within 1 second; `StackExchange.Redis` logs a successful connection

### AC-002: Hangfire SQL Server job store is registered and dashboard accessible to Admin only
- **Given**: The application is running and a user authenticates with an Admin JWT
- **When**: The user navigates to `GET /hangfire`
- **Then**: The Hangfire dashboard renders showing the job queue, succeeded, failed, and scheduled job tabs; no non-Admin user can access this URL (HTTP 403 returned)

### AC-003: Exponential backoff retry policy is registered for all background jobs
- **Given**: Hangfire is configured with the retry attribute
- **When**: A background job throws an exception on first attempt
- **Then**: Hangfire retries the job at approximately 30 s, 60 s, and 120 s (base 30 s, multiplier 2×, max 3 retries); after the third retry the job moves to the dead-letter (failed) queue

### AC-004: Cache-aside helper performs Redis GET before database query
- **Given**: A cache-aside helper method is called with a cache key that exists in Redis
- **When**: The helper executes
- **Then**: The Redis GET is called first; if the key is present the database query is not executed; if the key is absent the database is queried and the result is SET in Redis with the specified TTL

### AC-005: Redis unavailability does not crash the application
- **Given**: The Redis connection string points to an unreachable host
- **When**: The application starts and a cache-aside call is made
- **Then**: The `StackExchange.Redis` client throws a `RedisConnectionException`; the application catches it, falls through to the database query, logs a structured warning (not an unhandled exception), and continues serving requests

## Edge Cases
* Hangfire dead-letter queue grows unbounded — Admin dashboard should surface a count alert when failed jobs exceed a configurable threshold (default 10)
* Redis TTL race condition — two concurrent requests for the same uncached key should not result in duplicate database reads causing a thundering herd; implement a lightweight distributed lock or accept last-write-wins for TTL reset
* Hangfire job store SQL Server connection lost during job execution — Hangfire's in-built retry handles reconnection; job state is not lost
