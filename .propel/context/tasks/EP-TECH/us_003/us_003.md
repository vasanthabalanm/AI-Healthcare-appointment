# User Story - US_003

## Story ID
* US_003

## Story Title
* Dual-database EF Core setup SQL Server and PostgreSQL

## Description
* As a developer, I want two independent EF Core DbContext instances configured — `ApplicationDbContext` targeting SQL Server 2022 and `ClinicalDbContext` targeting PostgreSQL 16 — so that primary operational data and clinical data are isolated in separate databases with no shared transaction scope.

## Acceptance Criteria

### AC-001: ApplicationDbContext connects to SQL Server and migration runs clean
- **Given**: The `ConnectionStrings:SqlServer` environment variable points to a SQL Server 2022 Express instance
- **When**: `dotnet ef database update --context ApplicationDbContext` is executed
- **Then**: The `__EFMigrationsHistory` table is created in the SQL Server database and the migration completes with exit code 0

### AC-002: ClinicalDbContext connects to PostgreSQL and migration runs clean
- **Given**: The `ConnectionStrings:PostgreSql` environment variable points to a PostgreSQL 16 instance
- **When**: `dotnet ef database update --context ClinicalDbContext` is executed
- **Then**: The `__EFMigrationsHistory` table is created in the PostgreSQL database and the migration completes with exit code 0

### AC-003: The two contexts cannot share a transaction scope
- **Given**: A code path attempts to enlist both `ApplicationDbContext` and `ClinicalDbContext` in the same `System.Transactions.TransactionScope`
- **When**: The scope is opened and the second context is queried
- **Then**: An `InvalidOperationException` is thrown (or the design is enforced by a compile-time Roslyn analyser rule) confirming cross-context transactions are not permitted

### AC-004: Connection strings are read from environment variables, not appsettings
- **Given**: `appsettings.json` contains no connection string values
- **When**: The application starts with environment variables set
- **Then**: Both contexts connect successfully; if the environment variables are absent the application fails at startup with a descriptive configuration error

### AC-005: EF Core migration scaffolding target is explicit per context
- **Given**: `dotnet ef migrations add <Name>` is run without specifying `--context`
- **When**: The CLI outputs a warning
- **Then**: The warning instructs the developer to specify `--context ApplicationDbContext` or `--context ClinicalDbContext`; no ambiguous default migration is created

## Edge Cases
* SQL Server Express 10 GB database size limit approached — EF Core query plan should not be affected; a startup warning metric should be emitted when DB size exceeds 8 GB
* PostgreSQL connection pool exhaustion under high load — Npgsql max pool size configured and logged; connection timeout produces a structured log entry, not an unhandled exception
* Migration applied to wrong context — integration test harness verifies each entity type appears in the correct database by querying `information_schema.tables`
