# User Story - US_002

## Story ID
* US_002

## Story Title
* .NET 8 Web API vertical-slice scaffold with Swagger

## Description
* As a developer, I want the .NET 8 ASP.NET Core Web API project scaffolded in a vertical-slice feature folder architecture with Swashbuckle OpenAPI documentation and a health check endpoint, so that all feature slices can be added to a running, documented, and testable API host immediately.

## Acceptance Criteria

### AC-001: API project compiles and starts without errors
- **Given**: The solution is restored via `dotnet restore`
- **When**: `dotnet run` is executed from the API project directory
- **Then**: The process starts, binds to the configured HTTPS port, and outputs "Application started" to the console with zero compile errors

### AC-002: Swagger UI is accessible and documents all registered routes
- **Given**: The API is running in development mode
- **When**: A browser navigates to `GET /swagger`
- **Then**: The Swagger UI renders with the API title, version, and all registered controller routes listed; each route shows its HTTP verb, parameters, and response schemas derived from controller annotations

### AC-003: Vertical-slice feature folder structure is enforced
- **Given**: The project directory is inspected
- **When**: Listing the `Features/` directory
- **Then**: At least the stub directories `Auth/`, `Booking/`, `Intake/`, `Staff/`, `Documents/`, `Clinical/`, `Coding/`, `Audit/`, `Admin/` exist; each contains a placeholder `.cs` file to preserve the folder structure

### AC-004: Health check endpoint returns 200 with status payload
- **Given**: The API is running
- **When**: `GET /health` is called without authentication
- **Then**: The response is HTTP 200 with a JSON body `{"status":"Healthy"}` within 500 ms

### AC-005: Dependency injection container resolves without errors at startup
- **Given**: All services registered in `Program.cs`
- **When**: The application host builds (via `builder.Build()`)
- **Then**: No `InvalidOperationException` is thrown for unresolved dependencies; the DI container validates all registrations on startup in development mode

## Edge Cases
* DI circular dependency introduced by a developer — startup validation in development must surface the error immediately with the affected type names, not fail silently at runtime
* Port conflict on startup — application logs a clear "Address already in use" error and exits with non-zero code; does not hang
* OpenAPI generation fails for a malformed controller annotation — build pipeline fails; Swagger is not deployed with missing routes
