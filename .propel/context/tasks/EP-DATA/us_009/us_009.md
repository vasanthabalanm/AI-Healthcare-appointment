# User Story - US_009

## Story ID
* US_009

## Story Title
* ClinicalDocument entity schema and encrypted blob path

## Description
* As a developer, I want an EF Core migration for the ClinicalDocument entity in SQL Server, so that document metadata — including the encrypted blob storage path, virus scan result, and extraction status — is persisted with all required constraints while the encrypted binary content itself is stored outside the database.

## Acceptance Criteria

### AC-001: ClinicalDocument migration creates table with all required columns
- **Given**: The EF Core migration for ClinicalDocument is applied
- **When**: The SQL Server schema is inspected
- **Then**: The `ClinicalDocuments` table contains: `Id` (uniqueidentifier PK), `PatientId` (FK → UserAccounts), `OriginalFilename` (nvarchar NOT NULL), `MimeType` (nvarchar NOT NULL), `EncryptedBlobPath` (nvarchar(500) NOT NULL), `VirusScanResult` (int enum: Pending=0, Pass=1, Fail=2), `ExtractionStatus` (int enum: Pending=0, Extracted=1, LowConfidence=2, NoData=3, Failed=4), `UploaderId` (FK → UserAccounts), `UploadedAt` (datetime2 NOT NULL)

### AC-002: EncryptedBlobPath column stores only a filesystem path, not binary content
- **Given**: A ClinicalDocument is inserted with an `EncryptedBlobPath` value
- **When**: The row is read from SQL Server
- **Then**: The `EncryptedBlobPath` column contains a string path (e.g., `C:\clinical-docs\{guid}.enc`); no binary data or base64 content is stored in any database column

### AC-003: VirusScanResult defaults to Pending on INSERT
- **Given**: A new ClinicalDocument is inserted without specifying VirusScanResult
- **When**: The row is read from SQL Server
- **Then**: The `VirusScanResult` column value is `0` (Pending)

### AC-004: PatientId and UploaderId foreign keys reference UserAccounts with RESTRICT
- **Given**: The migration has been applied
- **When**: A DELETE is attempted on a UserAccount that owns ClinicalDocument rows
- **Then**: SQL Server raises a foreign key constraint violation; the UserAccount is not deleted

### AC-005: Index on PatientId enables efficient per-patient document queries
- **Given**: The migration includes a non-clustered index on `PatientId`
- **When**: A query `SELECT * FROM ClinicalDocuments WHERE PatientId = @id` is executed with SQL Server execution plan analysis
- **Then**: The execution plan shows an Index Seek (not a Table Scan) on the PatientId index

## Edge Cases
* EncryptedBlobPath refers to a file that no longer exists on disk — the database row remains valid; the application layer (EP-006) must handle file-not-found gracefully with a 404 and a structured log entry
* Two uploads race for the same GUID path — GUID generation in the application layer makes collision probability negligible; the unique PK on `Id` prevents duplicate metadata rows
* ExtractionStatus transition order — EF Core SaveChanges interceptor (implemented in EP-007) validates that status only advances forward (Pending → Extracted is valid; Extracted → Pending is not)
