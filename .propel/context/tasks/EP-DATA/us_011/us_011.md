# User Story - US_011

## Story ID
* US_011

## Story Title
* AuditLog append-only table, HIPAA PHI retention, and Redis TTL configuration

## Description
* As a developer, I want the AuditLog entity migration with INSERT-only database privileges, a soft-delete and PHI retention mechanism on all PHI-bearing entities, and Redis TTL constants configured in shared application settings, so that the platform satisfies HIPAA minimum retention requirements and enforces immutable audit records from the first deployment.

## Acceptance Criteria

### AC-001: AuditLog migration creates append-only table with INSERT-only SQL GRANT
- **Given**: The EF Core migration for AuditLog is applied and the SQL GRANT script runs
- **When**: The database role used by the application attempts `UPDATE audit_logs SET ... ` or `DELETE FROM audit_logs WHERE ...`
- **Then**: SQL Server raises a permission denied error (error 229); the UPDATE/DELETE never executes; the application user only holds INSERT and SELECT privileges on `audit_logs`

### AC-002: AuditLog table contains all required columns with auto-increment bigint PK
- **Given**: The migration is applied
- **When**: The SQL Server schema is inspected
- **Then**: The `AuditLogs` table contains: `Id` (bigint IDENTITY PK), `ActorId` (uniqueidentifier FK → UserAccounts nullable), `ActorRole` (int enum), `ActionType` (int enum: CREATE=0, READ=1, UPDATE=2, DELETE=3, LOGIN=4, LOGOUT=5, EXPORT=6), `EntityType` (nvarchar NOT NULL), `EntityId` (uniqueidentifier nullable), `BeforeValue` (nvarchar(MAX) nullable), `AfterValue` (nvarchar(MAX) nullable), `IpAddress` (nvarchar(45) NOT NULL), `Timestamp` (datetime2 NOT NULL DEFAULT GETUTCDATE())

### AC-003: Soft-delete and PHI retention columns exist on all PHI-bearing entities
- **Given**: Migrations for UserAccount, Appointment, IntakeRecord, ClinicalDocument, ExtractedClinicalField, and MedicalCodeSuggestion are inspected
- **When**: Each table schema is checked
- **Then**: All six tables contain `DeletedAt` (datetime2 nullable) and `RetainUntil` (datetime2 nullable); EF Core global query filter excludes rows where `DeletedAt IS NOT NULL` from default queries

### AC-004: EF Core SaveChanges override blocks permanent DELETE of PHI records within retention window
- **Given**: Application code calls `context.Remove(entity)` on a PHI-bearing entity within its 6-year retention window
- **When**: `SaveChanges` is called
- **Then**: The SaveChanges override converts the hard DELETE to a soft-delete (sets `DeletedAt = UtcNow`); no row is physically removed; a structured log warning is emitted

### AC-005: Redis TTL constants are centralised in shared configuration
- **Given**: A `CacheSettings` configuration class is registered in DI
- **When**: Any service reads the TTL for session tokens, slot availability, or 360° view
- **Then**: The values are read from `CacheSettings.SessionTtlSeconds = 900`, `CacheSettings.SlotTtlSeconds = 60`, `CacheSettings.View360TtlSeconds = 300`; no magic numbers appear in business logic code

## Edge Cases
* Attempt to set RetainUntil in the past — EF Core validation rejects a `RetainUntil` value earlier than `UtcNow + 1 day`; prevents accidental early expiry of HIPAA-protected records
* AuditLog insert failure during a transaction — the audit write must NOT be in the same `SaveChanges` call as the business entity write; it uses a separate DbContext instance to ensure audit entries are captured even if the business write is rolled back
* Redis TTL constant changed — requires a single config file update; no code changes required; configuration is validated at startup (negative or zero TTL raises an `OptionsValidationException`)
