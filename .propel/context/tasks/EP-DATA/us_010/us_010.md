# User Story - US_010

## Story ID
* US_010

## Story Title
* ExtractedClinicalField, ConflictFlag, and MedicalCodeSuggestion migrations

## Description
* As a developer, I want EF Core code-first migrations for ExtractedClinicalField, ConflictFlag, and MedicalCodeSuggestion entities in PostgreSQL, so that the clinical intelligence and AI coding data layer is schema-complete with all confidence score fields, FK relationships, and Trust-First NOT NULL constraints enforced at the database level.

## Acceptance Criteria

### AC-001: ExtractedClinicalField migration creates PostgreSQL table with all required columns
- **Given**: The EF Core migration is applied to PostgreSQL via ClinicalDbContext
- **When**: The PostgreSQL schema is inspected
- **Then**: The `extracted_clinical_fields` table contains: `id` (uuid PK), `document_id` (FK → clinical_documents via shadow reference or application-managed), `patient_id` (uuid NOT NULL), `field_type` (int enum: VitalSign=0, MedicalHistory=1, Medication=2, Allergy=3, Diagnosis=4), `field_value` (text NOT NULL), `confidence_score` (double precision NOT NULL), `extracted_at` (timestamptz NOT NULL)

### AC-002: ConflictFlag migration enforces two-field-reference structure
- **Given**: The EF Core migration for ConflictFlag is applied
- **When**: The PostgreSQL schema is inspected
- **Then**: The `conflict_flags` table contains: `id` (uuid PK), `field_id_1` (FK → extracted_clinical_fields), `field_id_2` (FK → extracted_clinical_fields), `resolution` (int enum: Unresolved=0, Resolved=1, Dismissed=2 NOT NULL DEFAULT 0), `resolved_by_id` (uuid nullable), `resolved_at` (timestamptz nullable)

### AC-003: MedicalCodeSuggestion enforces Trust-First NOT NULL check constraint
- **Given**: The migration for MedicalCodeSuggestion is applied
- **When**: An UPDATE is attempted that sets `status = 1` (Accepted) with `verified_by_id = NULL`
- **Then**: The database raises a check constraint violation (or the EF Core interceptor raises an exception before the write); the suggestion is not committed; the `verified_by_id` column remains null

### AC-004: MedicalCodeSuggestion contains all required columns and FK
- **Given**: The migration is applied
- **When**: The PostgreSQL schema is inspected
- **Then**: The `medical_code_suggestions` table contains: `id` (uuid PK), `patient_id` (uuid NOT NULL), `appointment_id` (uuid NOT NULL), `code_type` (int enum: ICD10=0, CPT=1), `suggested_code` (varchar(20) NOT NULL), `code_description` (text), `confidence_score` (double precision NOT NULL), `status` (int enum: Pending=0, Accepted=1, Modified=2, Rejected=3), `verified_by_id` (uuid nullable), `verified_at` (timestamptz nullable), `committed_code` (varchar(20) nullable)

### AC-005: Indexes on patient_id enable efficient per-patient clinical queries
- **Given**: Non-clustered indexes exist on `patient_id` in all three tables
- **When**: A query `SELECT * FROM extracted_clinical_fields WHERE patient_id = $1` is executed with PostgreSQL EXPLAIN ANALYZE
- **Then**: The execution plan shows an Index Scan (not a Sequential Scan) on the patient_id index

## Edge Cases
* ConflictFlag referencing the same field twice (field_id_1 = field_id_2) — a check constraint prevents self-referencing conflict flags
* MedicalCodeSuggestion `confidence_score` outside [0.0, 1.0] range — a check constraint `confidence_score BETWEEN 0.0 AND 1.0` enforces valid range
* PostgreSQL migration applied out of order — EF Core migration history table prevents replay; `dotnet ef database update` is idempotent
