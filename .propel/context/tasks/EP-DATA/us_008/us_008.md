# User Story - US_008

## Story ID
* US_008

## Story Title
* WaitlistEntry and IntakeRecord EF Core migrations

## Description
* As a developer, I want EF Core code-first migrations for the WaitlistEntry and IntakeRecord entities in SQL Server, so that waitlist management and versioned intake data have a fully constrained schema with uniqueness and referential integrity enforced at the database level.

## Acceptance Criteria

### AC-001: WaitlistEntry migration enforces one active entry per patient via partial unique index
- **Given**: The EF Core migration for WaitlistEntry is applied
- **When**: The SQL Server schema is inspected
- **Then**: The `WaitlistEntries` table exists with columns: `Id` (PK), `PatientId` (FK → UserAccounts), `PreferredSlotId` (FK → Slots), `ConfirmedAppointmentId` (FK → Appointments nullable), `Status` (int enum), `RegisteredAt` (datetime2); a filtered unique index exists on `(PatientId)` WHERE `Status = 0` (Active), preventing two Active entries per patient

### AC-002: WaitlistEntry Status enum values map correctly to database integers
- **Given**: A WaitlistEntry is saved with `Status = Active`
- **When**: The raw row is read from SQL Server
- **Then**: The `Status` column value is `0`; enum values map as Active=0, Offered=1, Accepted=2, Declined=3, Expired=4

### AC-003: IntakeRecord migration supports versioning with version number and editor FK
- **Given**: The EF Core migration for IntakeRecord is applied
- **When**: The SQL Server schema is inspected
- **Then**: The `IntakeRecords` table contains: `Id` (PK), `AppointmentId` (FK → Appointments), `Source` (int enum: AI=0, Manual=1), `StructuredFields` (nvarchar(MAX) for JSON), `Version` (int NOT NULL DEFAULT 1), `EditorId` (FK → UserAccounts nullable), `SubmittedAt` (datetime2 NOT NULL)

### AC-004: IntakeRecord query filter returns only the latest version by default
- **Given**: Three IntakeRecord rows exist for the same appointment with versions 1, 2, and 3
- **When**: The EF Core repository queries `IntakeRecords` for that appointment without explicit version filter
- **Then**: Only version 3 is returned; versions 1 and 2 are retrievable only with an explicit version filter or via the audit trail

### AC-005: Foreign key constraints use RESTRICT (no cascade delete)
- **Given**: A Patient UserAccount is the subject of a WaitlistEntry
- **When**: A DELETE is attempted on the UserAccount row directly in SQL Server
- **Then**: SQL Server raises a foreign key constraint violation; the account is not deleted; data integrity is preserved

## Edge Cases
* Partial unique index on WaitlistEntry — SQL Server 2022 Express supports filtered indexes; if the Express edition restriction affects index syntax, the migration script must be tested against Express specifically
* IntakeRecord JSON field size — `StructuredFields` stored as `nvarchar(MAX)`; EF Core value converter should validate JSON is well-formed before save; malformed JSON raises a `DbUpdateException` with a descriptive message
* Concurrent active waitlist insert for same patient — the filtered unique index ensures only one succeeds; the loser receives a constraint violation handled by the booking service
