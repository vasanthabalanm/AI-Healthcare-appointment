# User Story - US_007

## Story ID
* US_007

## Story Title
* UserAccount, Slot, and Appointment EF Core migrations

## Description
* As a developer, I want EF Core code-first migrations for the UserAccount, Slot, and Appointment entities in SQL Server, so that the primary user identity and appointment scheduling data layer is schema-complete with all constraints, indexes, and FSM status validation enforced at the database level.

## Acceptance Criteria

### AC-001: UserAccount migration creates table with all required columns and constraints
- **Given**: The EF Core migration for UserAccount is applied
- **When**: The SQL Server schema is inspected
- **Then**: The `UserAccounts` table contains: `Id` (uniqueidentifier PK), `FullName` (nvarchar NOT NULL), `DateOfBirth` (date NOT NULL), `Email` (nvarchar UNIQUE INDEX NOT NULL), `PasswordHash` (nvarchar NOT NULL), `Role` (int enum NOT NULL), `Status` (int enum NOT NULL), `EmailVerified` (bit NOT NULL), `CreatedAt` (datetime2 NOT NULL)

### AC-002: Slot migration creates table with rowversion for optimistic concurrency
- **Given**: The EF Core migration for Slot is applied
- **When**: The SQL Server schema is inspected
- **Then**: The `Slots` table contains: `Id` (uniqueidentifier PK), `SlotDate` (date NOT NULL), `SlotTime` (time NOT NULL), `Location` (nvarchar NOT NULL), `Capacity` (int NOT NULL), `IsAvailable` (bit NOT NULL), `RowVer` (rowversion NOT NULL); the `RowVer` column is configured as a concurrency token in EF Core

### AC-003: Appointment migration creates table with FK constraints and status enum
- **Given**: The EF Core migration for Appointment is applied
- **When**: The SQL Server schema is inspected
- **Then**: The `Appointments` table contains: `Id` (PK), `PatientId` (FK → UserAccounts.Id), `SlotId` (FK → Slots.Id), `Status` (int enum NOT NULL), `NoShowRiskScore` (int NOT NULL), `CancellationCutoff` (datetime2 NOT NULL), `CreatedAt`, `UpdatedAt`; FK constraints with ON DELETE RESTRICT (no cascade deletes on clinical data)

### AC-004: Appointment FSM status transition is validated in EF Core SaveChanges
- **Given**: An `Appointment` entity has status `Scheduled`
- **When**: Application code attempts to set status directly to `Completed` (skipping `Arrived` and `InProgress`)
- **Then**: The EF Core `SaveChanges` interceptor throws an `InvalidOperationException` listing the invalid transition; the database write does not occur

### AC-005: Email uniqueness is enforced at database level
- **Given**: A `UserAccount` row with `Email = 'test@example.com'` exists
- **When**: A second INSERT with the same email is attempted
- **Then**: SQL Server raises a unique constraint violation; EF Core translates this to a `DbUpdateException` with inner `SqlException` number 2601 or 2627

## Edge Cases
* Concurrent INSERT of two UserAccounts with same email — unique index prevents both from succeeding; one receives a constraint violation; no duplicate email can exist even under race conditions
* Slot rowversion mismatch during concurrent booking — EF Core `DbUpdateConcurrencyException` is thrown; the application layer (EP-002) handles retry/rejection; migration only ensures the column exists
* Migration rollback during deployment — `dotnet ef migrations script` generates an idempotent script that can be re-run safely
