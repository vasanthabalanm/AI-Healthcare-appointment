# User Story - US_029

## Story ID
* US_029

## Story Title
* AI conversational intake via Rasa NLU dialogue

## Description
* As a patient, I want to complete my pre-appointment intake by having a guided conversation with an AI assistant, so that I can provide my medical history and symptoms naturally instead of filling out a complex form, with the option to fall back to a manual form at any point.

## Acceptance Criteria

### AC-001: AI intake session starts and Rasa NLU responds with first dialogue turn
- **Given**: An authenticated patient calls `POST /intake/ai/start {appointmentId}`
- **When**: The request is forwarded to the Rasa service at `localhost:5005/webhooks/rest/webhook`
- **Then**: HTTP 200 is returned with `{"sessionId":"...","message":"<greeting from Rasa>","fields":[]}`; a session is created in Redis with TTL = 900 s

### AC-002: Dialogue turns extract structured fields incrementally
- **Given**: A patient sends a message via `POST /intake/ai/message {sessionId, text}`
- **When**: Rasa returns intent and entity extraction results
- **Then**: The extracted entities are appended to the session's `fields[]` accumulator; HTTP 200 is returned with `{"message":"<next Rasa prompt>","fields":[{"type":"VitalSign","value":"120/80","confidence":0.91}]}`

### AC-003: Patient confirms summary and IntakeRecord is created
- **Given**: Rasa signals `intent=confirm_summary` on the final dialogue turn
- **When**: The patient confirms via `POST /intake/ai/confirm {sessionId}`
- **Then**: HTTP 201 is returned; an `IntakeRecord` is inserted with `source=AI`, `version=1`, `structuredFields=[confirmed fields]`; the Redis session is deleted

### AC-004: Low-confidence entity triggers clarification prompt
- **Given**: Rasa returns an entity with `confidence < 0.70`
- **When**: The dialogue engine processes the extraction
- **Then**: The low-confidence entity is flagged and Rasa issues a clarification prompt for that entity; the field is not added to the confirmed fields accumulator until re-confirmed

### AC-005: Patient switches to manual intake mid-conversation
- **Given**: A patient calls `POST /intake/ai/switch-to-manual {sessionId}`
- **When**: The request is processed
- **Then**: HTTP 200 is returned with `{"message":"Switched to manual intake","fields":[...confirmed fields so far...]}`; already-confirmed fields are pre-populated in the manual form response; the Rasa session is terminated; the Redis session is deleted

### AC-006: Rasa service unreachable returns 503 with manual intake hint
- **Given**: The Rasa Windows Service at `localhost:5005` is not responding (timeout 5 s)
- **When**: Any AI intake endpoint is called
- **Then**: HTTP 503 is returned with `{"error":"AI intake temporarily unavailable","hint":"Use manual intake"}`; no patient data is lost; the error is logged with correlation ID

## Edge Cases
* Session timeout (Redis TTL=900s expires) — patient receives `{"error":"Intake session expired"}` on next message; they must restart the AI session; already-confirmed fields are not persisted
* Rasa returns no entities for an utterance — dialogue continues with Rasa's fallback prompt; no error is thrown
* Patient sends very long free-text — input truncated at 500 characters before forwarding to Rasa; patient is notified of the limit
