# User Story - US_030

## Story ID
* US_030

## Story Title
* Manual intake form submission

## Description
* As a patient, I want to submit my pre-appointment intake information using a structured form with clearly labelled fields, so that my medical history is recorded accurately when I prefer not to use the AI conversation or as a fallback when AI intake is unavailable.

## Acceptance Criteria

### AC-001: Valid form submission creates an IntakeRecord and returns 201
- **Given**: An authenticated patient submits `POST /intake/manual` with all required fields (chief complaint, medical history, current medications, allergies, relevant symptoms)
- **When**: The submission is processed
- **Then**: HTTP 201 is returned with `{"intakeId":"..."}`; an IntakeRecord is inserted with `source=Manual`, `version=1`, `appointmentId` linked, `submittedAt=UtcNow`

### AC-002: Missing required fields return 422 with field-level validation errors
- **Given**: `POST /intake/manual` is called with one or more required fields absent or blank
- **When**: The model validation runs
- **Then**: HTTP 422 is returned with `{"errors":{"chiefComplaint":["Required"],...}}`; no IntakeRecord is inserted; all field-level errors are returned in a single response

### AC-003: Pre-populated fields from AI session are preserved
- **Given**: A patient switched from AI intake mid-session with some confirmed fields
- **When**: `POST /intake/manual` is called with the pre-populated `fields` from the AI session included in the request body
- **Then**: The IntakeRecord includes the AI-confirmed fields plus any new manual-entry fields; `source=Manual` is recorded (overrides AI source once submitted manually)

### AC-004: Duplicate manual intake submission for same appointment returns 409
- **Given**: An IntakeRecord for the appointment already exists with `version=1`
- **When**: `POST /intake/manual` is called again for the same appointmentId
- **Then**: HTTP 409 is returned with `{"error":"Intake already submitted â€” use PATCH to update"}`; no new IntakeRecord is inserted

## Edge Cases
* Form submitted with plaintext SQL or script injection attempt in free-text fields â€” input sanitised by parameterised EF Core queries; no SQL execution risk; HTML-encoded in all API responses
* Patient submits intake after appointment time â€” the service checks the appointment status; if status is not Scheduled or Arrived, HTTP 409 is returned
* IntakeRecord structuredFields exceeds 10,000 characters â€” 422 returned with `{"errors":{"structuredFields":["Exceeds maximum length"]}}`
