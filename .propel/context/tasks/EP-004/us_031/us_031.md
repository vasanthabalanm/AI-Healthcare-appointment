# User Story - US_031

## Story ID
* US_031

## Story Title
* Intake data editing with version history

## Description
* As a Staff member, I want to edit a submitted intake record and have all prior versions preserved immutably, so that the patient's clinical history shows every revision with full before/after audit trail.

## Acceptance Criteria

### AC-001: PATCH creates a new IntakeRecord version, preserving the prior version
- **Given**: An IntakeRecord with `version=1` exists for an appointment
- **When**: `PATCH /intake/{appointmentId}` is called with updated fields
- **Then**: A new IntakeRecord row is inserted with `version=2`; the version 1 row is unchanged; HTTP 200 is returned with the new version

### AC-002: AuditLog entry records before/after state of the edit
- **Given**: A Staff member edits an IntakeRecord
- **When**: `PATCH /intake/{appointmentId}` is processed
- **Then**: An AuditLog entry is written with `actionType=UPDATE`, `entityType=IntakeRecord`, `entityId=appointmentId`, `beforeValue=<serialised version N fields>`, `afterValue=<serialised version N+1 fields>`, `actorId=staffId`

### AC-003: Default query returns only the latest version
- **Given**: IntakeRecords with versions 1, 2, and 3 exist for an appointment
- **When**: `GET /intake/{appointmentId}` is called without a version parameter
- **Then**: Only version 3 is returned; versions 1 and 2 are not included in the default response

### AC-004: Explicit version filter retrieves a prior version
- **Given**: Multiple versions exist for an intake record
- **When**: `GET /intake/{appointmentId}?version=1` is called
- **Then**: The version 1 IntakeRecord is returned with all its original field values

### AC-005: Patient cannot edit another patient's intake record
- **Given**: A JWT with `patientId=A` is used on `PATCH /intake/{appointmentId}` for an appointment belonging to `patientId=B`
- **When**: The request is processed
- **Then**: HTTP 403 is returned; an AuditLog entry records the unauthorised access attempt

## Edge Cases
* PATCH with no changes (fields identical to current version) — service detects no diff and returns HTTP 200 without creating a new version; AuditLog entry is NOT written for no-op edits
* Version counter reaches maximum int value — expected not to occur in production; if detected, a CRITICAL log alert is raised and the PATCH is rejected with 500 until resolved
* Concurrent edits to the same intake record — last-write-wins at version level; both create new version entries; no data is lost
