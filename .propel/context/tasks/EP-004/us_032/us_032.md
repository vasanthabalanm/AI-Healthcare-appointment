# User Story - US_032

## Story ID
* US_032

## Story Title
* Insurance pre-check soft validation at intake submission

## Description
* As a patient submitting intake, I want my insurance information to be validated non-blockingly against a reference table, so that I know whether my insurance details appear to be on file without my intake being rejected if a match is not found.

## Acceptance Criteria

### AC-001: Matching insurance details return Validated status without blocking intake
- **Given**: A patient submits intake with `insuranceProvider` and `policyNumber` that match a record in the InsuranceReference table
- **When**: `POST /intake/manual` includes insurance fields
- **Then**: The IntakeRecord is saved with `insuranceStatus=Validated`; intake submission proceeds to HTTP 201 regardless of insurance status; a non-blocking validation result is included in the response body

### AC-002: Non-matching insurance details return NotVerified without blocking intake
- **Given**: The `insuranceProvider` and `policyNumber` do not match any InsuranceReference record
- **When**: Intake is submitted
- **Then**: The IntakeRecord is saved with `insuranceStatus=NotVerified`; intake still returns HTTP 201; the response body includes `{"insuranceStatus":"NotVerified","hint":"Staff will review your insurance details before appointment"}`

### AC-003: Blank insurance fields return Skipped status
- **Given**: The patient submits intake with no insurance fields
- **When**: Intake is processed
- **Then**: `insuranceStatus=Skipped`; intake returns HTTP 201; no validation query is executed

### AC-004: Insurance validation result is visible to Staff on the patient record
- **Given**: An IntakeRecord has `insuranceStatus=NotVerified`
- **When**: Staff views the patient's intake summary
- **Then**: The insurance validation status is displayed; Staff can manually override the status to `Validated` after phone verification

## Edge Cases
* InsuranceReference table is unavailable — validation defaults to `NotVerified`; a structured log warning is emitted; intake submission succeeds
* Policy number with leading/trailing whitespace — normalised (trimmed and uppercased) before comparison
* Patient deliberately provides a wrong policy number — status is `NotVerified`; intake proceeds; no security risk as insurance status is advisory only
