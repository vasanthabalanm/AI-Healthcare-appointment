# User Story - US_044

## Story ID
* US_044

## Story Title
* 360° patient view assembly with Redis cache and Trust-First verification

## Description
* As a clinician, I want to view a complete 360° clinical summary of a patient — including demographics, appointment history, intake records, and extracted clinical fields — with a cache-backed response and a Staff verification mechanism, so that I have a reliable, fast, and auditable view of the patient's full record.

## Acceptance Criteria

### AC-001: 360° view cache hit returns response from Redis within 500ms
- **Given**: A 360° view has been assembled and cached in Redis as `view360:{patientId}` with TTL=300s
- **When**: `GET /patients/{id}/view360` is called
- **Then**: HTTP 200 is returned with the cached view; no PostgreSQL query is executed; response arrives within 500 ms

### AC-002: Cache miss triggers PostgreSQL assembly and populates Redis cache
- **Given**: No cached 360° view exists in Redis for the patient
- **When**: `GET /patients/{id}/view360` is called
- **Then**: PostgreSQL queries retrieve demographics, appointments, intake records, and active non-duplicate ExtractedClinicalFields; the assembled view is SET in Redis with EX 300; HTTP 200 is returned

### AC-003: Staff verification sets patient record status to Verified
- **Given**: No unresolved ConflictFlags exist for the patient
- **When**: `PATCH /patients/{id}/verify {staffId}` is called by a Staff member
- **Then**: HTTP 200 is returned; the patient's clinical record `verificationStatus` is set to `Verified`; `verifiedById=staffId`, `verifiedAt=UtcNow`; the Redis 360° cache is invalidated; an AuditLog entry is written

### AC-004: Verification attempt with unresolved conflicts returns 409
- **Given**: The patient has ConflictFlags with `resolution=Unresolved`
- **When**: `PATCH /patients/{id}/verify` is called
- **Then**: HTTP 409 is returned with `{"error":"Resolve all clinical conflicts first","unresolvedCount":N}`; no status change is made

### AC-005: Patient with no documents has 360° view with an informational hint
- **Given**: A patient has no uploaded ClinicalDocuments
- **When**: `GET /patients/{id}/view360` is called
- **Then**: HTTP 200 is returned with `{"demographics":{...},"appointments":[...],"clinicalFields":[],"hint":"No clinical documents uploaded yet"}`; HTTP 404 is not returned

## Edge Cases
* Redis unavailable during 360° view request — falls through to direct PostgreSQL query without caching; a structured warning is logged; HTTP 200 is still returned
* Patient record verified but new document uploaded after verification — upload resets `verificationStatus` to `Unverified`; the 360° cache is invalidated; Staff must re-verify
* 360° view assembly takes > 2 seconds — a structured warning is logged; if > 5 seconds, an alert is raised and the result is cached regardless to prevent repeated slow queries
