# User Story - US_042

## Story ID
* US_042

## Story Title
* De-duplication of extracted clinical fields via Hangfire aggregation job

## Description
* As a clinician, I want duplicate extracted clinical fields from multiple document uploads to be automatically de-duplicated, so that the patient's 360° view shows clean, non-redundant clinical data.

## Acceptance Criteria

### AC-001: Aggregation job runs after each field-extraction job completes
- **Given**: A Hangfire field-extraction job completes successfully for a patient
- **When**: The extraction job's continuation callback fires
- **Then**: A Hangfire aggregation job is enqueued for the same `patientId`; the job is not enqueued if no new fields were extracted

### AC-002: De-duplication retains the most recent non-conflicting field per field type per source context
- **Given**: A patient has two `Medication` fields extracted from different documents: `Aspirin 100mg` (extracted 3 days ago) and `Aspirin 100mg` (extracted today)
- **When**: The aggregation job runs `DeduplicateByFieldType`
- **Then**: The older duplicate `Aspirin 100mg` field is soft-deleted (marked `isDuplicate=true`); only the most recent is retained in the active view

### AC-003: Conflicting values are not de-duplicated — a ConflictFlag is inserted instead
- **Given**: A patient has two `Medication` fields: `Metformin 500mg` and `Metformin 1000mg`
- **When**: The aggregation job processes these fields
- **Then**: Both fields are retained; a `ConflictFlag` row is inserted referencing both field IDs with `resolution=Unresolved`; neither field is soft-deleted

### AC-004: Redis 360° view cache is invalidated after aggregation completes
- **Given**: A cached 360° view exists in Redis for the patient
- **When**: The aggregation job completes
- **Then**: Redis `DEL view360:{patientId}` is called; the next GET triggers a fresh assembly of the 360° view from PostgreSQL

## Edge Cases
* Aggregation job runs while field-extraction job for the same patient is still in progress — the aggregation job uses a patient-scoped distributed lock; it waits up to 10 seconds and then fails gracefully with a reschedule
* All fields for a patient are de-duplicated leaving zero active fields — the 360° view returns an empty fields array with a "No clinical data available" message rather than an error
* `isDuplicate` flag incorrectly set — field can be manually re-activated by Staff via a PATCH endpoint; the AuditLog records the reactivation
