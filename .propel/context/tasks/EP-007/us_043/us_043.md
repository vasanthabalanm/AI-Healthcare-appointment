# User Story - US_043

## Story ID
* US_043

## Story Title
* Clinical data conflict detection and Staff resolution

## Description
* As a clinician, I want conflicting extracted clinical fields flagged automatically and presented to me for resolution, so that the 360° patient view only contains verified, non-contradictory data.

## Acceptance Criteria

### AC-001: Conflicting clinical fields trigger ConflictFlag insertion during aggregation
- **Given**: The aggregation job detects two ExtractedClinicalField rows for the same patient with the same `fieldType` but different `fieldValue` (e.g., two different blood pressure readings)
- **When**: The aggregation job processes the fields
- **Then**: A `ConflictFlag` row is inserted with `field_id_1`, `field_id_2`, `resolution=Unresolved`; neither field is de-duplicated; the patient's 360° view assembly includes a `hasConflicts=true` flag

### AC-002: Staff resolves a conflict by accepting one field value
- **Given**: A ConflictFlag with `resolution=Unresolved` exists
- **When**: `PATCH /conflicts/{id}/resolve {acceptedFieldId, staffId}` is called
- **Then**: `ConflictFlag.resolution` is set to `Resolved`; `resolved_by_id = staffId`; `resolved_at = UtcNow`; the rejected field's `isDuplicate` is set to `true`; the Redis 360° cache is invalidated; an AuditLog entry is written

### AC-003: Staff dismisses a conflict without choosing a winner
- **Given**: A ConflictFlag with `resolution=Unresolved` exists
- **When**: `PATCH /conflicts/{id}/dismiss {staffId, reason}` is called
- **Then**: `ConflictFlag.resolution` is set to `Dismissed`; both fields remain active in the 360° view; the dismiss reason is stored; AuditLog is updated

### AC-004: Attempting to verify a 360° view with unresolved conflicts returns 409
- **Given**: A patient's record has one or more ConflictFlags with `resolution=Unresolved`
- **When**: `PATCH /patients/{id}/verify` is called (Trust-First verification)
- **Then**: HTTP 409 is returned with `{"error":"Resolve all clinical conflicts before verification","unresolvedCount":2}`

## Edge Cases
* Same conflict detected twice (e.g., same two field IDs conflict again after re-extraction) — duplicate ConflictFlag detection checks for an existing unresolved flag with the same field pair; if found, no duplicate is inserted
* Field referenced by a ConflictFlag is manually deleted by Staff — the ConflictFlag should be auto-dismissed; a foreign key set-null or application-level cleanup job handles orphaned flags
* Concurrent Staff resolving the same conflict — EF Core rowversion prevents double-resolution; the second Staff receives HTTP 409 with a refresh hint
