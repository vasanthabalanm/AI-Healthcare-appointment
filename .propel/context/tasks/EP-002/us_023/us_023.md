# User Story - US_023

## Story ID
* US_023

## Story Title
* Patient cancels or reschedules an appointment

## Description
* As a patient, I want to cancel or reschedule my upcoming appointment up to a configurable cutoff time before it is due, so that the released slot becomes available to other patients and I receive a confirmation of the change.

## Acceptance Criteria

### AC-001: Cancellation within allowed window succeeds and releases slot
- **Given**: The current time is before the appointment's `cancellationCutoff`
- **When**: `DELETE /appointments/{id}` is called by the owning patient
- **Then**: HTTP 200 is returned; Appointment status is set to `Cancelled`; Slot `isAvailable` is set to `true`; pending reminder Hangfire jobs for this appointment are cancelled; a slot-swap check job is triggered; a cancellation confirmation email is sent

### AC-002: Cancellation after cutoff returns 409
- **Given**: The current time is after the appointment's `cancellationCutoff`
- **When**: `DELETE /appointments/{id}` is called
- **Then**: HTTP 409 is returned with `{"error":"Cancellation cutoff reached — contact Staff"}`; no appointment or slot change is made

### AC-003: Reschedule within allowed window moves the appointment to a new slot
- **Given**: The current time is before `cancellationCutoff` and the new slot is available
- **When**: `PATCH /appointments/{id}/reschedule {newSlotId}` is called
- **Then**: HTTP 200 is returned; old slot `isAvailable` is set to `true`; new slot `isAvailable` is set to `false` (with rowversion lock); Appointment `slotId` is updated; Redis cache is invalidated for both slot dates; a swap-check job is triggered for the old slot

### AC-004: Patient cannot cancel another patient's appointment
- **Given**: A JWT with `patientId=A` is used to cancel `appointmentId` belonging to `patientId=B`
- **When**: `DELETE /appointments/{id}` is called
- **Then**: HTTP 403 is returned; no change is made; an AuditLog entry records the unauthorised attempt

### AC-005: Reschedule to an already-taken slot returns 409
- **Given**: The target `newSlotId` has `isAvailable=false`
- **When**: `PATCH /appointments/{id}/reschedule {newSlotId}` is called
- **Then**: HTTP 409 is returned with `{"error":"Slot no longer available"}`; the original appointment is unchanged

## Edge Cases
* Appointment already in `Cancelled` status — cancelling a cancelled appointment returns 409 `{"error":"Appointment is already cancelled"}`
* Reschedule to same slot ID — treated as a no-op; HTTP 200 returned with no database changes
* Cutoff time falls on a weekend or holiday — cutoff is always stored as an absolute UTC timestamp at booking time; no business-day logic adjustments
