# User Story - US_021

## Story ID
* US_021

## Story Title
* Automated preferred-slot swap via Hangfire SwapMonitor job

## Description
* As a patient, I want to be automatically offered my preferred slot when it becomes available and have my appointment moved if I accept within the 2-hour response window, so that I get the appointment time I actually want without constant manual checking.

## Acceptance Criteria

### AC-001: Slot release triggers SwapMonitor job and offer is sent
- **Given**: A patient cancels or reschedules, releasing a slot that is the preferred slot of an Active waitlist entry
- **When**: The cancellation completes and the SwapMonitor Hangfire job is triggered
- **Then**: The first patient in the waitlist ordered by `registeredAt ASC` receives an offer email and SMS; WaitlistEntry status is updated to `Offered`; `offeredAt` is set to UtcNow

### AC-002: Patient accepts swap within 2-hour window â€” appointment is moved
- **Given**: A WaitlistEntry is in `Offered` status within the 2-hour response window
- **When**: `POST /waitlist/{id}/accept` is called by the patient
- **Then**: The new slot's `isAvailable` is set to false; the old slot's `isAvailable` is set to true; the Appointment's `slotId` is updated to the preferred slot; WaitlistEntry status is set to `Accepted`; a confirmation email is sent; the Redis slot cache is invalidated for both slot dates

### AC-003: Slot taken by another booking between offer and acceptance â€” swap fails gracefully
- **Given**: A WaitlistEntry is in `Offered` status and the preferred slot is concurrently booked by another patient
- **When**: `POST /waitlist/{id}/accept` is called
- **Then**: The EF Core transaction detects the slot concurrency conflict; the swap is rolled back; WaitlistEntry status is set to `Expired`; the patient receives a notification "Preferred slot no longer available"; HTTP 409 is returned

### AC-004: Patient declines swap within window
- **Given**: A WaitlistEntry is in `Offered` status
- **When**: `POST /waitlist/{id}/decline` is called
- **Then**: HTTP 200 is returned; WaitlistEntry status is set to `Declined`; the slot remains available; the next waitlisted patient (if any) is offered the slot

### AC-005: Response window expires without patient action
- **Given**: A WaitlistEntry has `status=Offered` and `offeredAt` is more than 2 hours ago
- **When**: The Hangfire expiry job runs its scheduled check
- **Then**: WaitlistEntry status is set to `Expired`; the slot is offered to the next eligible patient; the expired patient is notified

## Edge Cases
* No waitlisted patient exists for the released slot â€” SwapMonitor job exits immediately without side effects; slot becomes generally available
* Patient's confirmed appointment was already cancelled before swap acceptance â€” acceptance validates the confirmed appointment still exists and is Scheduled; if not, swap is aborted with a clear error
* Configurable 2-hour window is changed to a different value â€” the response window is read from `AppSettings.SwapOfferWindowHours`; Hangfire expiry job respects the configured value
