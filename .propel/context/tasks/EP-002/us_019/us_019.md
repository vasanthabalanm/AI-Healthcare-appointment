# User Story - US_019

## Story ID
* US_019

## Story Title
* Patient browses slots and books appointment with optimistic lock

## Description
* As a patient, I want to view available appointment slots and book one with a confirmation, so that I can secure my appointment knowing the slot is reserved exclusively for me even under concurrent booking conditions.

## Acceptance Criteria

### AC-001: Available slots returned from Redis cache within 60 seconds TTL
- **Given**: Slot availability for a date has been cached in Redis
- **When**: `GET /slots?date=2026-06-01` is called
- **Then**: HTTP 200 is returned with the slot list; Redis `GET slots:date:2026-06-01` is the data source; no SQL Server query is executed; response arrives within 500 ms

### AC-002: Cache miss falls through to SQL Server and populates cache
- **Given**: No Redis cache entry exists for the requested date
- **When**: `GET /slots?date=2026-06-01` is called
- **Then**: SQL Server is queried for `isAvailable=true` slots; results are SET in Redis with TTL 60 seconds; HTTP 200 is returned with slot list

### AC-003: Successful booking creates appointment and invalidates slot cache
- **Given**: A patient is authenticated and the requested slot is available
- **When**: `POST /appointments {slotId, patientId}` is called
- **Then**: HTTP 201 is returned with `{"appointmentId":"..."}`; a Slot row is updated to `isAvailable=false`; an Appointment row is inserted with `status=Scheduled` and a calculated `noShowRiskScore`; the Redis slot cache entry for that date is deleted; confirmation email and reminder Hangfire jobs are enqueued

### AC-004: Concurrent booking of same slot ā€” second request receives 409
- **Given**: Two patients simultaneously call `POST /appointments` for the same slotId
- **When**: Both requests reach the booking service concurrently
- **Then**: Exactly one receives HTTP 201; the other receives HTTP 409 `{"error":"Slot no longer available"}`; the Slot rowversion mismatch triggers a `DbUpdateConcurrencyException` for the losing request

### AC-005: Patient with existing active appointment receives 409 on new booking
- **Given**: A patient has an existing Appointment with `status=Scheduled`
- **When**: `POST /appointments {slotId, patientId}` is called for a different slot
- **Then**: HTTP 409 is returned with `{"error":"Cancel existing appointment first"}`; no new appointment is created

## Edge Cases
* Slot capacity is 0 ā€” `isAvailable` is set to `false` at migration; booking endpoint checks `isAvailable` before rowversion lock; 409 returned immediately without beginning a transaction
* Booking with a past slot date ā€” 400 returned with `{"error":"Slot date must be in the future"}`
* Redis cache entry missing after successful booking (cache invalidation failed) ā€” next GET for that date triggers a cache miss and re-populates from SQL Server; no stale data served
