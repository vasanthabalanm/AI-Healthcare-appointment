# User Story - US_047

## Story ID
* US_047

## Story Title
* Trust-First Staff code verification interface

## Description
* As a Staff member, I want to review, verify, modify, or reject AI-suggested medical codes one by one or in bulk, and have no AI-generated code committed to the clinical record without my explicit sign-off, so that every submitted code has a named human accountable for its accuracy.

## Acceptance Criteria

### AC-001: Staff retrieves pending code suggestions for a patient
- **Given**: `MedicalCodeSuggestion` rows with `status=Pending` exist for a patient
- **When**: `GET /patients/{id}/code-suggestions` is called by Staff
- **Then**: HTTP 200 is returned with a list of `{suggestionId, codeType, suggestedCode, codeDescription, confidenceScore, lowConfidenceFlag, status}`; suggestions are grouped by `codeType` (ICD-10 and CPT separately)

### AC-002: Staff accepts a suggestion with verifiedById — returns 200
- **Given**: A suggestion with `status=Pending` exists
- **When**: `PATCH /code-suggestions/{id} {status:"Accepted", verifiedById:"<staffId>"}` is called
- **Then**: HTTP 200 is returned; `status=Accepted`, `verifiedById=staffId`, `verifiedAt=UtcNow`, `committedCode=suggestedCode`; an AuditLog entry is written

### AC-003: PATCH without verifiedById returns 422 (Trust-First enforcement)
- **Given**: Any status change is attempted for a suggestion
- **When**: `PATCH /code-suggestions/{id} {status:"Accepted"}` is called without a `verifiedById` field
- **Then**: HTTP 422 is returned with `{"error":"verifiedById is required for code verification"}`; the suggestion status is not changed

### AC-004: Bulk accept-all with verifiedById marks all Pending suggestions as Accepted
- **Given**: Multiple `Pending` suggestions exist for a patient
- **When**: `POST /patients/{id}/code-suggestions/accept-all {verifiedById:"<staffId>"}` is called
- **Then**: All `Pending` suggestions for the patient are updated to `Accepted` with `verifiedById` and `verifiedAt`; AuditLog entries are written for each; HTTP 200 is returned with count of accepted suggestions

### AC-005: Coding-complete gate blocks completion if any Pending suggestions remain
- **Given**: One or more suggestions still have `status=Pending`
- **When**: `POST /patients/{id}/coding-complete` is called
- **Then**: HTTP 409 is returned with `{"error":"Coding incomplete — pending suggestions remain","pendingCount":N}`; coding is not marked complete

### AC-006: AuditLog entry is written for every individual verification action
- **Given**: Staff verifies, modifies, or rejects a suggestion
- **When**: Any `PATCH /code-suggestions/{id}` call is processed
- **Then**: An AuditLog entry is written with `entityType=MedicalCodeSuggestion`, `entityId=suggestionId`, `actorId=staffId`, `beforeValue`, `afterValue`

## Edge Cases
* Staff modifies the suggested code (changes from AI code to a different code) — `PATCH` includes `committedCode` field; the modified code is stored in `committedCode` (not `suggestedCode`); this counts as a "Modified" action in agreement-rate metrics
* Suggestion already Accepted — a second PATCH with `status=Accepted` returns HTTP 409 `{"error":"Suggestion already verified"}`
* verifiedById references a non-existent Staff user — FK validation rejects the update; HTTP 422 returned
