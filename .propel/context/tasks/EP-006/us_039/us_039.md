# User Story - US_039

## Story ID
* US_039

## Story Title
* AES-256 document encryption at rest and role-gated secure retrieval

## Description
* As a compliance officer, I want all clinical documents encrypted with AES-256-CBC before any disk write, and retrieval gated to authorised roles only, so that stored documents cannot be read by any user who obtains raw filesystem access.

## Acceptance Criteria

### AC-001: Document is encrypted with AES-256-CBC before acknowledgement is returned
- **Given**: A valid uploaded file has passed the ClamAV scan
- **When**: The encryption step runs using `System.Security.Cryptography.Aes.Create()`
- **Then**: The file is written to disk in encrypted form before HTTP 201 is returned to the caller; no plaintext version of the file exists at any point on disk or in any application buffer after the encryption write completes

### AC-002: Encryption key is loaded from environment variables or Windows DPAPI — never from config files
- **Given**: The application starts
- **When**: The document encryption service resolves the AES key
- **Then**: The key is read from the `CLINICAL_AES_KEY` environment variable or Windows DPAPI; no AES key material exists in any committed `appsettings.json`, `.env`, or source file; application startup fails with a CRITICAL log if the key is missing

### AC-003: Secure document retrieval is restricted to Staff and Admin roles
- **Given**: A Patient JWT calls `GET /documents/{id}/download`
- **When**: The RBAC policy is evaluated
- **Then**: HTTP 403 is returned; no decrypted content is returned to the Patient role; the access attempt is logged in AuditLog

### AC-004: Authorised Staff retrieves document — decrypted content is streamed, not written to temp file
- **Given**: A Staff JWT calls `GET /documents/{id}/download`
- **When**: The document is retrieved
- **Then**: The encrypted file is read from disk, decrypted in-memory using the AES key, and streamed directly as `application/pdf` in the HTTP response body; no decrypted file is written to any temporary path on the server filesystem

### AC-005: Tampered or corrupted encrypted file returns 422
- **Given**: The encrypted blob file has been modified on disk (e.g., bit-flipped)
- **When**: `GET /documents/{id}/download` is attempted
- **Then**: AES decryption fails with a `CryptographicException`; HTTP 422 is returned with `{"error":"Document integrity check failed"}`; the corruption event is logged as CRITICAL with the document ID

## Edge Cases
* AES key rotated in environment — existing files were encrypted with the previous key; key rotation requires a migration job to re-encrypt all blobs with the new key; this is a planned admin operation documented in the runbook
* In-memory decryption buffer for a large file — a 10 MB file is retained in memory for the duration of the HTTP response stream; the buffer is disposed immediately after the stream is flushed
* Simultaneous download requests for the same document — each request decrypts independently; no shared decrypted buffer exists
